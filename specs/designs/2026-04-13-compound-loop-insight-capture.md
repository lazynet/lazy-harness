# Compound-loop Insight Capture

**Date:** 2026-04-13
**Status:** Proposed — deferred
**Related backlog:** `compound-loop: learnings lost on long sessions` (shares delta-by-index mechanism)

## Problem

Insights generated by the assistant mid-conversation (e.g. `★ Insight ─` blocks from the `explanatory` output style) frequently die before reaching persistent memory. Three concrete loss paths, all observed in the 2026-04-13 session:

1. **Gate-out of short sessions.** `process_task()` in `compound_loop.py:476` rejects any session with `<200 user chars` or `<4 messages`. A dense single-question session that produces valuable insights is discarded whole.
2. **Tail window in long sessions.** `extract_messages()` passes only the last 20 messages to the headless-Claude extractor. Insights surfaced early in a long session fall out of the window before the worker sees them.
3. **Lossy semantic classification.** The extractor prompt targets `decisions / failures / learnings / handoff` and rewrites findings into 1-2 sentence summaries. Verbatim insight blocks — which often encode dense, specific codebase knowledge — get paraphrased or dropped as "not universal enough" to qualify as `learnings`.

The net effect: the very content the `explanatory` output style was designed to produce is the content least likely to survive the pipeline.

## Goal

Make `★ Insight ─` blocks first-class citizens of the compound-loop pipeline: captured verbatim, gate-independent, dedupe-safe, and deterministic (regex, not LLM), while still integrating cleanly with the existing extractor so the LLM step does not duplicate them.

## Non-goals

- General curation UX for the insight store.
- Back-filling insights from historical JSONL files already on disk.
- Replacing the existing `learnings / decisions / failures / handoff` extraction. This is additive.
- Generating insights where the assistant did not author one. No LLM-generated paraphrasing.

## Design

### 1. Regex extraction, pre-LLM

Add `extract_insights(session_jsonl: Path, since_index: int = 0) -> list[Insight]` to `compound_loop.py`. It:

- Streams the JSONL.
- For each message with `role == "assistant"` and `index >= since_index`, scans the text for blocks matching roughly `★ Insight ─+\n(.*?)\n─+` (multiline, non-greedy). The exact regex should tolerate the Unicode box-drawing characters used by the output style.
- Returns a list of `Insight` dataclasses carrying `body`, `message_index`, `timestamp`, `session_id`, and a `content_hash` (sha256 of the body with whitespace normalized).
- Ignores `role == "user"` matches entirely — only the assistant can author a canonical insight.

This step runs before any gate check and before any call to `claude -p`.

### 2. Gate bypass when insights are present

In `process_task()`, compute `insights = extract_insights(...)` first. If `insights` is non-empty, skip the `min_user_chars` and `min_messages` gates for the rest of the task: the session has dense content even if it is short or one-sided. Log the bypass explicitly (`bypass: N insights found`) so the reason is visible in `compound-loop.log`.

The gates still apply when no insights are found — they exist for good reason (noise suppression), and bypassing them unconditionally would reintroduce the noise problem.

### 3. Deterministic persistence

Write each insight to `memory/insights/YYYY-MM/YYYY-MM-DD-<session_id_short>-<n>.md` with frontmatter:

```yaml
---
session_id: <full>
message_index: <int>
timestamp: <iso8601>
source: assistant
content_hash: <sha256-prefix>
---
```

Followed by the verbatim body. Persistence happens **before** the LLM extractor runs — so insights survive even if `claude -p` times out or the JSON parse fails. This is the core durability guarantee: regex-found insights never depend on the LLM step succeeding.

### 4. Deduplication

Before writing, compute `content_hash` and check against an in-memory set built from existing files under `memory/insights/**`. Skip writes whose hash already exists. This makes the extraction idempotent across re-runs (Stop hook fires multiple times on the same session, which we see today in the logs).

### 5. LLM prompt integration

Pass the discovered insights into `build_prompt()` as a new block parallel to the existing `## Existing learnings already recorded` section:

```
## Insights already captured verbatim from this session (DO NOT re-emit as learnings):
- <title or first line of each>
```

The extractor is then free to produce `learnings` that build on or generalize the insights, but will not duplicate their content. This keeps the semantic layer (LLM) and the deterministic layer (regex) complementary rather than competing.

### 6. Delta tracking (ties into existing backlog)

Store `last_insight_message_index` per session in the existing per-session state used by the worker's idempotency check. Subsequent Stop hooks on the same session re-scan only the delta (`since_index = last + 1`). This is the same delta-by-index mechanism proposed in the backlog item `compound-loop: learnings lost on long sessions`; implementing it here means the fix for both issues lands together.

## Test plan (TDD — red first)

Mandatory ordering: each test written and watched to fail before the code that makes it pass.

1. `test_extract_insights_parses_single_block` — one assistant message with one `★ Insight ─` block returns one `Insight` with the correct body and index.
2. `test_extract_insights_parses_multiple_blocks_per_message` — one message containing two non-adjacent blocks returns two insights.
3. `test_extract_insights_ignores_user_messages` — identical marker in a user message returns nothing.
4. `test_extract_insights_respects_since_index` — messages before `since_index` are skipped.
5. `test_persist_insights_writes_verbatim_with_frontmatter` — output file matches fixture, frontmatter fields populated.
6. `test_persist_insights_is_idempotent_by_hash` — running twice over the same transcript writes files only once.
7. `test_process_task_bypasses_gate_when_insights_present` — 71-char session + one insight block → bypass logged, persistence happens, no `skipped` outcome.
8. `test_process_task_still_gates_empty_short_session` — 71-char session, zero insights → still skipped with the existing message.
9. `test_build_prompt_includes_captured_insights_block` — when insights are passed in, the prompt contains the "DO NOT re-emit" section.
10. `test_insights_persisted_even_if_claude_invocation_fails` — stub `invoke_claude` to return `None`; insight files must still exist on disk.
11. `test_delta_scan_skips_already_processed_indices` — second run of the same session only scans messages after `last_insight_message_index`.

## Open questions

- **Storage location.** `memory/insights/` under the project memory dir, or a global `~/.claude-lazy/insights/`? Project-local matches the current `learnings` layout and keeps insights scoped to the repo they were generated in. Recommended: project-local.
- **Regex robustness.** The `★` and `─` characters come from the output style template. If the template ever changes, the regex breaks silently. Mitigation: add a contract test that pins the exact markers the current `explanatory` style emits, so a template change forces a visible failure.
- **mkdocs nav.** Whether this spec and future ones are public. Resolved: specs live in `specs/` (not published), not in `docs/`. This file moved from the orphaned `docs/specs/` location as part of the roadmap PR.
