# ADR-009: Profiles deploy as symlinks from the harness config dir into each agent config dir

**Status:** accepted
**Date:** 2026-04-13

## Context

A profile is a complete, self-contained agent configuration — `CLAUDE.md`, `skills/`, any additional instructions, and eventually a `settings.json` populated by the framework. Users typically maintain one or more profiles (`personal`, `work`, `flex`, etc.) to isolate unrelated contexts.

The framework needs to get that content into the exact directory Claude Code reads from. Claude Code reads from `~/.claude/` by default, or from whatever `CLAUDE_CONFIG_DIR` points at. A single user with multiple profiles therefore needs multiple target directories — `~/.claude-personal/`, `~/.claude-work/`, etc. — and Claude Code must be launched with the right `CLAUDE_CONFIG_DIR` per session.

Two additional constraints:

- The source of truth must remain inside `~/.config/lazy-harness/profiles/<name>/`, owned by the user, versioned with their dotfiles. The framework must not own or rewrite that content.
- Iterating on a profile (editing `CLAUDE.md`, adding a skill) must be instantaneous — no "rebuild" step, no copy-on-write that silently drifts from source.

## Decision

**Deploy profiles as symlinks, per file, from `~/.config/lazy-harness/profiles/<name>/*` into `<profile.config_dir>/*`.**

Flow implemented in `src/lazy_harness/deploy/engine.py`:

1. Source: `~/.config/lazy-harness/profiles/<name>/` (a real directory the user owns).
2. Target: `expand_path(profile_entry.config_dir)` — typically `~/.claude-<name>/`. The target is read from `config.toml`'s `[profiles.<name>]` section.
3. For each entry inside the source profile directory, `ensure_symlink` creates `target_dir/entry_name -> source/entry_name`. Existing symlinks that already point at the correct source are a no-op (status `"exists"`).
4. The default profile additionally gets a top-level symlink: `~/.claude -> <target_dir>` via `deploy_claude_symlink`, so that plain `claude` invocations without `CLAUDE_CONFIG_DIR` still land on the default profile.

Corollary decisions:

- **Hook configuration is written, not symlinked.** `deploy_hooks` generates a `settings.json` with the fully-resolved hook commands and writes it into each profile's target dir. This file is framework-owned output; the user is not expected to edit it.
- **Per-file symlinks, not a whole-directory symlink.** `lh deploy` symlinks each item inside the profile source, so a `settings.json` generated by the framework can coexist with user-authored files like `CLAUDE.md` and `skills/` in the same target dir without the user's content becoming a single opaque link.
- **`CLAUDE_CONFIG_DIR` is how you select a profile per session.** Profile resolution (`src/lazy_harness/core/profiles.py`) matches the current directory against each profile's `roots` list; longest match wins, with a `default` fallback.

## Alternatives considered

- **Copy profile content into the target dir on deploy.** Breaks the "instant iteration" constraint: every edit to `CLAUDE.md` would need a `lh deploy` step. Also turns source and target into two places that can drift.
- **Hardlinks instead of symlinks.** Work on same-filesystem targets, but fail across filesystem boundaries (common when the user keeps dotfiles on an external volume or encrypted disk) and do not survive moving the source directory.
- **Bind mounts.** Linux-only; macOS has nothing equivalent short of `bindfs`. Not portable.
- **Point `CLAUDE_CONFIG_DIR` directly at `~/.config/lazy-harness/profiles/<name>/`.** Tempting — no deploy step at all. Rejected because Claude Code wants to write session state, cache, and `settings.json` into its config dir, and we do not want the user's dotfile-versioned profile directory to receive that write traffic. The symlink hop keeps write-side state in the target dir (e.g. `~/.claude-personal/projects/`) and leaves the source dir read-only from Claude Code's perspective.
- **Git worktree per profile.** Gives per-profile isolation for free but forces every user onto a specific dotfile-management workflow. Rejected; the framework must be unopinionated about dotfile tooling ([ADR-001](001-hybrid-architecture.md)).

## Consequences

- Editing a profile is atomic: change `CLAUDE.md` in the source directory, the target already reflects it via the symlink. No deploy step needed for content edits.
- `lh deploy` is idempotent. Re-running it on an already-deployed profile just verifies each symlink and reports "already linked". Used by the selftest to assert the deploy is healthy.
- Claude Code's write-side state (`projects/`, memory files, session JSONLs) lands in the target dir, separate from the user's version-controlled source. This is why the `context-inject` hook computes paths like `<CLAUDE_CONFIG_DIR>/projects/<encoded-cwd>/memory/` — the memory is owned by the deployed side, not the source side.
- A broken profile is easy to debug: `ls -la ~/.claude-<name>/` shows every symlink, and `readlink` shows the source path directly.
- Adding a profile is two commands: create the source directory under `~/.config/lazy-harness/profiles/<name>/`, then `lh profile add <name> --config-dir ~/.claude-<name>` to register it. `lh deploy` links the content.
- On Windows, NTFS symlinks require a privilege that many setups do not grant by default. This is a known gap; the framework will likely grow a `--strategy=copy` flag for that platform rather than invent a third mechanism.
